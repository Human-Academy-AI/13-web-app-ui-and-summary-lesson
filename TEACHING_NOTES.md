# TEACHING_NOTES

## 現時点の合意（これまでの議論の反映）
- 教材は段階学習にする: `UIライブラリなし` → `Bootstrap` → `Tailwind + DaisyUI`
- 第1段階では「HTML/CSSだけでも完成度を上げられる」体験を優先する
- 第2段階では「UIライブラリを明示指定すると品質と速度が上がる」ことを比較体験する
- 仕様書には非機能要件と教材意図を分離して記載し、受講者が読み取りやすい構成にする
- 既存の翻訳ロジックは維持し、UI改善は見た目と操作性にフォーカスして進める

## 教材設計メモ（意図）
- 設計を人が主導し、AIは実装加速に使う役割分担を体験させる
- 仕様の書き方で成果物品質が変わることを実感させる
- UI改善を「色・余白・タイポグラフィ・情報配置」の観点で説明できる状態を目指す

## 仕様の割り切り（教材用）
- UIライブラリなし版はデスクトップ優先（モバイル最適化は必須にしない）
- Bootstrap版とTailwind + DaisyUI版はレスポンシブ対応を目標にする
- バックエンドや翻訳モデルの高度化は本回の主目的に含めない

## 学習目標（到達イメージ）
- 5W1Hで仕様を具体化できる
- 仕様書 → UI設計 → 実装 の一連の流れを説明できる
- 同一要件で複数UIアプローチを比較し、採用理由を言語化できる
- AIへの指示精度（制約・優先順位・完成条件）が品質に直結することを理解する

## 各フェーズのねらいと観察ポイント
### 1) UIライブラリなし（現在のカスタムUI）
- ねらい
  - 素のHTML/CSSでも十分に整ったUIを作れることを体験する
  - 「どの情報をどこに置くか」の設計思考を強める
- 観察ポイント
  - 見出し、入力欄、操作ボタン、補助テキストの視線誘導が自然か
  - 翻訳元/翻訳先の操作手順が迷いなく伝わるか
  - 文言（例: API未連携時の注記）が学習者に誤解を生まないか

### 2) Bootstrap版
- ねらい
  - コンポーネント活用で一貫性と実装速度が上がる体験を得る
  - レスポンシブ調整を短時間で行う方法を学ぶ
- 観察ポイント
  - 仕様書でBootstrap利用を明示したときの出力安定性
  - グリッド/フォーム/ボタンの統一感

### 3) Tailwind + DaisyUI版
- ねらい
  - 「CSSを手で書く」から「ユーティリティクラスを組む」発想へ移行する
  - デザイン試行回数を増やし、比較検討の速度を上げる
- 観察ポイント
  - クラス設計の可読性
  - DaisyUIテーマ利用時の配色・コントラストの妥当性

## 授業進行メモ（実施時の流れ）
- まず受講者に仕様書を書かせる（制約と完成条件を必ず記述）
- 次に同じ要件でUIライブラリなし版を作り、見た目の改善余地を洗い出す
- その後Bootstrap版またはTailwind + DaisyUI版へ展開し、差分比較を行う
- 最後に「なぜその指示で結果が変わったか」を振り返り、仕様書を改訂させる

## 評価観点（チェックリスト）
- 要件が曖昧語ではなく観察可能な表現で書かれている
- UIの情報階層（タイトル・操作・結果）が明確
- 入力から結果表示までの導線が短く、迷いが少ない
- ライブラリ採用の有無によるメリット/デメリットを説明できる
- AI出力をそのまま採用せず、意図に沿って修正できている

## 次回に向けた拡張候補
- エラーハンドリング表示（入力空、API失敗時）をUI課題として追加
- モバイル最適化を比較課題として追加
- 配色アクセシビリティ（コントラスト比）を評価項目に追加
